Technical Blog 5 - Problem-Solving

Discussing with a non-tech friend about a time when you:

  1. Got blocked on a simple problem
  2. Solved a problem in an elegant way

In both cases:

  • What was the problem?

    • I was blocked on assignment 5.4 where we had to write a program which turn the dots blue when you click on it.

  • What problem solving techniques did you use?

    • I asked code-help-desk on slack for help. Also asked my husband James, who has a more logical brain than me. Someone on Slack said I should put 'click' instead of 'contextmenu'. I had been copying the code from before without really understanding where I needed to change. James helped by suggesting I use a javascript validator (it's like spell-check for programmers) and it picked up a rogue ( as well as the fact there are no ; at the end of each statement.

  • How did you feel throughout the process?

    • Really frustrated initially. Was starting to think this is not for me. What am I doing here? Why did I spent most of our savings on a course that my brain was simply not wired for? Still a big hit on my self-confidence, even today.

  • What did you learn?

    • Use a validator when writing. I'm not good at picking up rogue typos and there are programs that do that for me. Using console.log to check what your code does at each step of the way. I should have done the 5.5 Debugging assignment before doing 5.4. I learn that I should just move on to the next assignment if I can't figure out the answer to a current assignment.

Relfect on how confident you feel using the problem solving techniques and process:

  • Pseudocode

    • Not very confident with pseudocode. I think given the open texture of the English language, I could never write a sufficiently unambiguous pseudocode to help. I'd much prefer just commenting what I want to do next.

  • Trying something

    • This is something that I've been doing and feels like I'm leaving everything to chance when I do it. I keep a record of the different configurations I've tried, like in legal research.

  • Rubber ducky method

    • I have a small soft toy dugon (sea cow) which my friend bought me from Sydney Aquarium. Its arms hang on the top of the laptop so I can talk to it. Hasn't really worked as I feel like a crazy person talking out loud by myself!

  • Reading error messages

    • Most of the time I don't understand what the error messages say so I'm not very confident with this technique.

  • Console.logging

    • I've discovered this technique recently. If only EDA taught me to console.log at each step before the assignment!

  • Googling

    • I'd have to say this is the technique I'm most confident in. Sometimes however, the answer is given to me when Googling and I feel that I've cheated. I would make sure I understand why the answer works to solve the problem. An example is the check board exercise - using # sign to make a check board.

  • Asking your peers for help

    • I've only just started using this technique on slack. People are very friendly and are eager to help. I really like how friendly most coders are.

  • Asking coaches for help

    • Usually I will try this if all of the above has been exhausted. I felt terrible when Harrison said he is looking after about 60 phase 0 students. I hope the teacher-student ratio is not as bad when we get to bootcamp!

  • Improving process with reflection

    • Haven't really been doing this given that we are 'sprinting' but I know there are things I could do differently if I get to start again!
